# 樂高組合模型 — Workshop 開發方法論

> 服務是可重用的積木。專案是積木的組裝。不區分「專案」和「模組」——只有服務和組合。

---

## 雙線夾擊

Workshop 的開發模型同時由上而下與由下而上推進：

```
由上而下（需求 → 藍圖 → 待辦清單）
    │
    ▼         ┌───────────────────────────────┐
    ├────────►│  匯合點：積木能力               │
    │         │  ≥ 需求 = 可以出貨              │
    ▲         └───────────────────────────────┘
    │
由下而上（打造可重用的服務積木）
```

### 由下而上：打造服務積木

每個服務都是一塊樂高積木，具備：
- 清晰的**能力邊界**（finance 處理金錢、quest 處理任務、matching 處理配對）
- 清晰的 **API 介面**（REST 端點 + MCP 工具 + 事件類型）
- 清晰的**成長路徑**（漸進式複雜度 — 每個階段增加新能力）

### 由上而下：分析需求

當新需求到來時（「我想要一個法律顧問」、「我想要一個 POS 系統」）：
1. 分析需要哪些能力
2. 盤點哪些現有服務已經涵蓋
3. 設計藍圖：需要擴展什麼、需要新增什麼
4. 拆解為待辦項目

### 匯合點：複利效應

每個新專案都會擴展現有服務的能力：
- 為法律顧問擴展 memory → RAG 能力強化了 memory 本身
- 為虛擬客服擴展 matching → 配對引擎變得更通用

服務越多且越成熟，新專案組裝得越快——這就是**複利效應**。

## 組合配方

> 「專案」的本質：挑選服務積木 → 擴展所需能力 → 組裝成專用解決方案。

### 配方一：法律顧問

```
memory ──► 法律文件 RAG（司法文書嵌入）
intel  ──► 判例搜尋（司法資料庫 + 法規資料庫）
muse   ──► 策略彙編（法律論證知識圖譜）
media  ──► 紙本文件 OCR
```

| 服務 | 需擴展的能力 |
|------|-------------|
| memory | 法律文件嵌入管線（基於司法資料庫的 RAG） |
| intel | 司法院裁判 / 全國法規資料庫連接器 |
| muse | 策略彙編範本、法律論證知識圖譜 |
| media | 法律文件 OCR（紙本 → 數位） |

**預期流程**：
1. 輸入案件詳情 → intel 搜尋相關判例和法條
2. memory 透過 RAG 檢索過往類似分析
3. muse 建構策略知識圖譜（論點 + 反論點）
4. 模擬庭審：LLM 根據所有輸入進行推理
5. 輸出：策略彙編文件

**資料來源**：台灣司法院裁判查詢系統、全國法規資料庫

---

### 配方二：教會音樂數位化

```
media  ──► 樂譜 OCR（紙本 → MusicXML/MIDI）
memory ──► 音樂後設資料建檔（筆畫數、調性、拍號）
muse   ──► 曲庫管理（索引、搜尋）
media  ──► 音訊合成（伴奏 + 人聲）
```

| 服務 | 需擴展的能力 |
|------|-------------|
| media | 樂譜 OCR → MusicXML/MIDI 輸出 |
| media | 音訊合成（MIDI → 伴奏、旋律+歌詞 → 人聲） |
| memory | 音樂後設資料索引（筆畫數、調性、拍號、原始出處） |
| muse | 曲庫管理、搜尋/瀏覽介面 |

**預期流程**：
1. 掃描紙本樂譜 → media（樂譜 OCR） → 數位格式
2. 存入 memory 並附帶後設資料
3. 透過 muse 曲庫瀏覽/搜尋
4. 透過 media 產生伴奏 / 合成人聲
5. 人工審核：自動產生 → 人工編輯 → 發佈

---

### 配方三：虛擬客服

```
media        ──► 產品目錄 OCR（紙本 → 結構化資料庫）
matching     ──► 需求 × 產品目錄 → 排序建議
social-hooks ──► LINE Bot 作為客戶端前台
quest        ──► 客戶詢問 → 自動建立任務追蹤
finance      ──► 報價單產生（PDF/HTML）
```

| 服務 | 需擴展的能力 |
|------|-------------|
| media | 產品目錄 OCR → 結構化資料庫 |
| matching | 產品目錄 × 需求配對引擎 |
| quest | 客戶詢問 → 自動建立任務追蹤 |
| finance | 報價單產生（PDF/HTML） |
| social-hooks | LINE Bot 客服對話流程 |

---

### 配方四：ERP / POS

```
finance    ──► 庫存管理 + POS 交易流程
quest      ──► 訂單生命週期（報價 → 接單 → 出貨）
workforce  ──► 機台/設備資源追蹤
matching   ──► 資源分配最佳化
```

| 服務 | 需擴展的能力 |
|------|-------------|
| finance | 庫存管理、POS 交易流程、庫存追蹤 |
| quest | 訂單生命週期（報價 → 接單 → 出貨 → 開票） |
| workforce | 機台/設備資源追蹤、維護排程 |
| matching | 資源分配最佳化（人力 + 機台） |

**成長路徑**：這是 finance + quest + workforce 各自成熟到 Phase 4 時的自然匯合點。

## 決策流程：新需求到來時該怎麼做？

```
新需求到來
    │
    ▼
盤點：需要哪些能力？
    │
    ├──► 現有服務已涵蓋 → 直接組裝，撰寫組合配方
    │
    ├──► 部分涵蓋 → 擴展現有服務能力 + 組裝
    │
    └──► 全新領域 → 評估是否值得打造新服務積木
              │
              ├──► 值得（會被多個配方重用） → 打造新服務
              └──► 不值得（一次性需求） → Station 或 Script
```
