# 軟體工程原則與設計模式參考指南

  I. 五大基礎原則 (SOLID)

  由 Robert C. Martin (Uncle Bob) 提出，為物件導向設計的核心：

  原則：S — Single Responsibility
  全稱：Single Responsibility (SRP)
  一句話概括：一個類別只做一件事，且只有一個理由會使其改變
  實作範例：finance-svc 只處理財務資訊，絕不涉及 auth
  ────────────────────────────────────────
  原則：O — Open/Closed
  全稱：Open/Closed (OCP)
  一句話概括：對擴展開放，對修改封閉（在不更動現有代碼的情況下增加功能）
  實作範例：Plugin 架構：在不修改 Core 的情況下增加新的 Plugins
  ────────────────────────────────────────
  原則：L — Liskov Substitution
  全稱：Liskov Substitution (LSP)
  一句話概括：子類別可以完全替換父類別
  實作範例：所有 MCP 伺服器都遵循相同的協定
  ────────────────────────────────────────
  原則：I — Interface Segregation
  全稱：Interface Segregation (ISP)
  一句話概括：不要強迫實作不使用的介面
  實作範例：不要讓 media-mcp 實作 finance 介面
  ────────────────────────────────────────
  原則：D — Dependency Inversion
  全稱：Dependency Inversion (DIP)
  一句話概括：高層模組不應依賴低層模組——兩者都應依賴抽象
  實作範例：Service 依賴於 Repository 介面，而非直接依賴於 PostgreSQL

  II. 一般開發原則

  原則：DRY
  全稱：Don't Repeat Yourself
  一句話概括：同一套邏輯只寫一次——重複時應進行提取
  ────────────────────────────────────────
  原則：KISS
  全稱：Keep It Simple, Stupid
  一句話概括：若能保持簡單，就不要使其複雜化
  ────────────────────────────────────────
  原則：YAGNI
  全稱：You Aren't Gonna Need It
  一句話概括：不要為了「以後可能需要」而開發
  ────────────────────────────────────────
  原則：SSOT
  全稱：Single Source of Truth
  一句話概括：每一項數據都應有且僅有一個權威來源
  ────────────────────────────────────────
  原則：MVP
  全稱：Minimum Viable Product
  一句話概括：能驗證假設的最簡版本
  ────────────────────────────────────────
  原則：WET
  全稱：Write Everything Twice
  一句話概括：DRY 的反面——有時重複好過錯誤的抽象（三則原則：重複三次再進行抽象）
  ────────────────────────────────────────
  原則：LoD
  全稱：Law of Demeter (Principle of Least Knowledge)
  一句話概括：只與直接的朋友交談——避免像 a.b.c.d() 這樣的鏈式調用
  ────────────────────────────────────────
  原則：CoC
  全稱：Convention over Configuration
  一句話概括：遵循框架慣例，減少配置撰寫
  ────────────────────────────────────────
  原則：SoC
  全稱：Separation of Concerns
  一句話概括：不同的職責不應混雜在一起
  ────────────────────────────────────────
  原則：PIE
  全稱：Program to an Interface, not an Implementation
  一句話概括：針對介面編寫代碼，以降低耦合
  ────────────────────────────────────────
  原則：CQS
  全稱：Command-Query Separation
  一句話概括：讀取操作 (Query) 不改變狀態；寫入操作 (Command) 不回傳數據
  ────────────────────────────────────────
  原則：CQRS
  全稱：Command Query Responsibility Segregation
  一句話概括：架構層級的 CQS——將讀取和寫入分離到不同的模型或數據庫
  ────────────────────────────────────────
  原則：PoLA
  全稱：Principle of Least Astonishment
  一句話概括：行為應符合使用者預期
  ────────────────────────────────────────
  原則：Fail Fast
  全稱：-
  一句話概括：儘早暴露錯誤——絕不靜默地吞掉它們
  ────────────────────────────────────────
  原則：Defensive Programming
  全稱：-
  一句話概括：假設所有輸入都是不可信的——驗證所有邊界
  ────────────────────────────────────────
  原則：Boy Scout Rule
  全稱：-
  一句話概括：離開時營地要比發現時更乾淨（每次接觸代碼時都改進一點點）
  ────────────────────────────────────────
  原則：Composition > Inheritance
  全稱：-
  一句話概括：優先使用組合而非繼承來複用行為
  ────────────────────────────────────────
  原則：Premature Optimization
  全稱：-
  一句話概括：「過早優化是萬惡之源」(Knuth) ——先求可行，再求速度

  III. 架構層級原則

  原則：Loose Coupling
  描述：最小化模組間的依賴——更動 A 不應影響 B
  ────────────────────────────────────────
  原則：High Cohesion
  描述：模組內的內容應高度相關
  ────────────────────────────────────────
  原則：Bounded Context
  描述：DDD 核心——每個領域都有自己的「語言」與邊界
  ────────────────────────────────────────
  原則：Ubiquitous Language
  描述：團隊與代碼使用同一套術語 (DDD)
  ────────────────────────────────────────
  原則：Event-Driven
  描述：模組間透過事件溝通，而非直接調用
  ────────────────────────────────────────
  原則：Idempotency
  描述：執行相同操作多次所產生的結果，應與執行一次相同
  ────────────────────────────────────────
  原則：CAP Theorem
  描述：分散式系統在一致性 (Consistency)、可用性 (Availability)、分區容錯性 (Partition tolerance) 中只能三選二
  ────────────────────────────────────────
  原則：Eventually Consistent
  描述：以即時一致性換取保證所有節點最終同步
  ────────────────────────────────────────
  原則：12-Factor App
  描述：雲原生應用的 12 條原則（配置存於環境變數、無狀態等）
  ────────────────────────────────────────
  原則：Strangler Fig
  描述：逐步替換舊系統，而非一次性重寫
  ────────────────────────────────────────
  原則：Anti-Corruption Layer
  描述：新舊系統間的轉換層——防止舊設計污染新架構

  IV. 設計模式 (GoF 23 + 現代補充)

  建立型 — 如何建立物件

  模式：Singleton
  一句話概括：全域僅有一個實體
  常見用途：DB 連線池、配置資訊
  ────────────────────────────────────────
  模式：Factory Method
  一句話概括：父類別定義介面，子類別決定建立內容
  常見用途：create_mcp_server(type)
  ────────────────────────────────────────
  模式：Abstract Factory
  一句話概括：建立相關物件的家族
  常見用途：UI 主題工廠
  ────────────────────────────────────────
  模式：Builder
  一句話概括：逐步建構複雜物件
  常見用途：複雜查詢建構器
  ────────────────────────────────────────
  模式：Prototype
  一句話概括：透過複製現有物件來建立新物件
  常見用途：深度複製配置資訊
