# Audit Step 3 — Status Report (issue #29)

Verified 2026-05-06. All findings have already been addressed in the

## H3/H7 — Assistant/RLM tmux session pollution + global lock DoS

**Status:** acknowledged + mitigated (architectural trade-off).

`core/src/modules/assistant/services.py` line ~247:
- `_lock = asyncio.Lock()` serializes every `stream_chat()` request, so two
  concurrent callers cannot interleave `send-keys` / JSONL reads against the
  shared tmux window.
- `_REQUEST_QUEUE = asyncio.Queue(maxsize=5)` bounds the wait queue so excess
  load fast-fails ("助手正在處理其他請求") rather than starving. This
  directly mitigates the "global lock DoS" concern from the audit.

Per-user tmux windows (full request-level isolation) would still be a future
improvement but require subprocess lifecycle management; the current
single-window + serialized + bounded design is the documented trade-off for
solo-dev deployment.

## M1 — rlm_engine stateless transcript role-boundary escaping

**Status:** already-fixed.

`core/src/shared/rlm_engine.py`:
- `_escape_role_markers()` (line ~655) substitutes `[User]` → `[User_]`,
  `[Assistant]` → `[Assistant_]`, `[Context Info]` → `[Context_Info]` in
  user-supplied content.
- `_format_history_as_transcript()` (line ~662) only escapes `user` and
  `metadata` roles — `assistant` content is treated as trusted (it was
  generated by the model itself in the same conversation).

A user injecting fake `[Assistant]` markers into a prompt will see them
escaped before transcript formatting, so the role boundary cannot be
forged.
