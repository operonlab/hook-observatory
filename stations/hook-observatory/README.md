# hook-observatory

![Python 3.12+](https://img.shields.io/badge/python-3.12%2B-blue)
![License: MIT](https://img.shields.io/badge/license-MIT-green)
![Claude Code](https://img.shields.io/badge/claude--code-compatible-purple)

A centralized, configurable hook handler framework for [Claude Code](https://docs.anthropic.com/en/docs/claude-code). Replaces scattered shell scripts with a single Python dispatcher, 34 handler modules, and YAML-based configuration.

---

## Why?

Claude Code's hook system is powerful — every tool call, session event, and agent action can be intercepted. But as hooks accumulate, you end up with dozens of shell scripts wired into `settings.json`, each re-implementing the same boilerplate: JSON parsing, error handling, timeout management, config loading.

hook-observatory solves this with a single dispatcher that routes all 10 event types through a unified handler registry, gives you a 5-second time-budgeted execution model, and lets you toggle any handler on or off from one YAML file.

---

## Quick Start

```bash
# 1. Clone or place this directory anywhere
cd path/to/hook-observatory

# 2. Preview changes without touching anything
python3 install.py --dry-run

# 3. Install
python3 install.py

# 4. Copy and edit config
# (install.py does this automatically, but you can do it manually)
cp config.example.yaml config.yaml

# 5. Restart Claude Code
```

That's it. All 10 Claude Code hook events now route through the dispatcher.

## Architecture

```
~/.claude/settings.json        (10 event types)
         │
         ▼
~/.claude/hooks/dispatcher.py  (thin shim — auto-generated by install.py)
         │
         ▼
handlers/__init__.py           (registry + 2-phase dispatch)
         │
    ┌────┴─────┐
    ▼          ▼
Critical    Deferrable
(always)    (5s budget)
```

### Two-phase execution

Every event dispatch runs handlers in two phases:

**Phase 1 — Critical handlers** run unconditionally with no time limit. These are safety and security gates that must never be skipped.

**Phase 2 — Deferrable handlers** share a 5-second budget. If the budget is exhausted, remaining handlers are skipped gracefully (logged, not blocked). This prevents slow integration handlers from delaying Claude.

The final decision merges all handler results: `block` wins over `approve`, which wins over passthrough.

### Fail-open guarantee

Every handler is wrapped in a try/except. A crashing handler returns `ALLOW` — hook-observatory never blocks Claude Code due to a bug in a handler.

## Handlers

### Core handlers (enabled by default)

No external dependencies. Safe to enable for any Claude Code installation.

| Handler | Event | What it does |
|---------|-------|--------------|
| `bash_safety` | PreToolUse:Bash | Blocks dangerous commands: `rm -rf /`, `sudo`, `git push --force`, bare `npm`/`yarn` instead of `pnpm` |
| `auto_format` | PostToolUse:Edit/Write | Runs `ruff` on Python files, `biome` on JS/TS after edits |
| `secret_scan` | PreToolUse:Bash | Scans staged diff for hardcoded secrets before `git push` |
| `agent_naming` | PreToolUse:Agent | Enforces descriptive `name` parameter on Agent tool calls |
| `skill_security` | PreToolUse:Write/Edit | Blocks prompt injection patterns in skill files |
| `verify_commit` | PreToolUse:Bash | Gates `git commit` until a test-verification marker is present |
| `verify_completion` | SubagentStop | Gates sub-agent completion until tests pass |
| `review_gate` | Stop | Warns about uncommitted changes before Claude stops |
| `plan_impl_gate` | PostToolUse:ExitPlanMode + UserPromptSubmit | Reminds to save plan to memory after plan mode exits |
| `observability` | All events | Spools event logs for audit and debugging |

Critical handlers (always run regardless of budget): `bash_safety`, `secret_scan`, `agent_naming`, `skill_security`, `verify_commit`, `review_gate`.

### Workflow handlers

Useful quality-of-life handlers. Some require `gh` CLI or `tmux`.

| Handler | Event | What it does |
|---------|-------|--------------|
| `context_inject` | SubagentStart | Injects context into sub-agent prompts |
| `context_relay` | SessionStart + PreCompact | Relays session context across compactions |
| `claudemd_suggest` | SessionStart | Suggests CLAUDE.md improvements based on session patterns |
| `cleanup_versions` | SessionStart | Cleans up stale versioned files in the project |
| `utility_watchdog` | SessionStart + SessionEnd | Monitors handler utility scores |
| `pm_autopilot` | PostToolUse:Bash + Stop | Syncs commits to GitHub Issues, suggests closing merged issues |
| `rtk_rewrite` | PreToolUse:Bash | Rewrites commands for token efficiency (requires `rtk` binary) |

### Integration handlers (disabled by default)

Require external services. Enable in `config.yaml` after setting up the corresponding service.

| Handler | Requires | What it does |
|---------|----------|--------------|
| `voice_notify` | TTS service or `edge-tts` | Speaks notifications aloud at session stop/start |
| `sentinel_notify` | Sentinel station | Notifies Sentinel on Bash events for health tracking |
| `anvil_telemetry` | Anvil station | Emits structured telemetry for all events |
| `session_channel` | Session Channel station | Publishes session events to Redis Streams |
| `memory_sync` | Memvault service | Auto-extracts memories from edited skill files |
| `session_pipeline` | Session Pipeline service | Runs session archival pipeline on SessionEnd |
| `session_namer` | LLM API (Haiku) | Auto-renames sessions based on conversation content |
| `session_cost` | None (standalone JSONL) | Tracks per-session response counts |
| `instinct_distiller` | Instinct pipeline | Distills behavioral patterns from session history |
| `attitude_signal` | Corrections pipeline | Tracks correction signals from Notification events |
| `relay_signal` | `tmux` | Signals tmux panes on session stop |
| `schedule_sync` | `schedules/` directory | Syncs schedule manifests after edits |
| `external` | Memvault scripts | Handles recall injection, skill tracking, progressive extraction |

## Configuration

Copy `config.example.yaml` to `config.yaml` (gitignored) and edit:

```yaml
# Toggle handlers on/off
handlers:
  core:
    bash_safety: true
    auto_format: true
    secret_scan: true
    # ... etc

  integrations:
    voice_notify: false        # enable after setting up TTS
    anvil_telemetry: false     # enable after setting up Anvil
    # ... etc

# Tool paths — "auto" finds them in PATH
tools:
  python: auto
  ruff: auto
  biome: auto

# Service URLs (only matter if handler is enabled)
services:
  tts_url: "http://127.0.0.1:10201/api/tts/speak"
  anvil_url: "http://127.0.0.1:10301"

# Time budget for deferrable handlers (ms)
dispatcher:
  blocking_budget_ms: 5000
```

A `config.local.yaml` file (also gitignored) can override any value from `config.yaml` — useful for machine-specific paths or per-project overrides.

## Installation

```bash
# Standard install (auto-detects Python 3.12+)
python3 install.py

# Explicit Python interpreter
python3 install.py --python ~/.local/bin/python3

# Preview without making changes
python3 install.py --dry-run

# Remove all hooks from settings.json
python3 install.py --uninstall
```

The installer:
1. Detects or accepts a Python interpreter path
2. Writes `~/.claude/hooks/dispatcher.py` (the thin shim)
3. Registers all 10 event types in `~/.claude/settings.json`
4. Creates `config.yaml` from `config.example.yaml` if it doesn't exist

Restart Claude Code after installing.

## Escape hatches

Every blocking hook has a bypass for emergencies:

| Hook | Bypass |
|------|--------|
| `bash_safety` (pnpm enforcement) | `PNPM_LOCK_DISABLE=1 claude` |
| `secret_scan` | `SECRET_SCAN_DISABLE=1 claude` or add `# nosec` comment to the line |
| `verify_commit` | Set the verification marker in the commit message |
| Any handler | Set `handler_name: false` in `config.yaml` and restart |

## Writing a custom handler

Create a Python file in `handlers/` with a `handle` function:

```python
from .base import ALLOW, HookResult, block, message

def handle(event_type: str, tool_name: str, tool_input: dict, raw_input: str) -> HookResult:
    # Return ALLOW to pass through
    if tool_name != "Bash":
        return ALLOW

    command = tool_input.get("command", "")
    if "dangerous-pattern" in command:
        return block("Blocked: reason shown to Claude")

    return message("Optional informational message")
```

Then register it in `handlers/__init__.py` in the `REGISTRY` dict under the relevant event type, and add it to `config.example.yaml` under the appropriate group.

### HookResult reference

| Constructor | Decision | Use for |
|-------------|----------|---------|
| `ALLOW` | passthrough | Default — let Claude proceed |
| `block("reason")` | block | Prevent the tool call |
| `approve()` | approve | Explicitly approve (skips other checks) |
| `message("msg")` | passthrough | Show a message without blocking |
| `text_result("text")` | passthrough | Return raw text (UserPromptSubmit only) |

### Design principles

**Block-at-Submit, not Block-at-Write.** Blocking `Edit`/`Write` operations confuses the agent mid-task. Block at `git commit` or `git push` time instead, when the intent is clear.

**Fail-open.** Every handler must catch its own exceptions. A handler that raises an unhandled exception returns `ALLOW` automatically — this is enforced by the dispatcher, but handlers should still be defensive.

**Escape hatch for every blocking rule.** If a hook can block a valid workflow, provide an environment variable or annotation to bypass it.
