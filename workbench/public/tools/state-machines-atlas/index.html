理論表達能力

Chomsky 階梯 ★★☆☆☆ 理論基礎

按計算能力排序：能辨識什麼語言？

以形式語言理論為基礎，根據「能記多少、能算多少」排列四種自動機。從正則語言到可計算理論上限。

粒子沿台階邊緣往上爬 — 下層無法突破上層邊界

成員

DFA
Deterministic Finite Automaton
確定性 FSM，每個 (state, input) 只有一個 next state
NFA
Non-deterministic FA
非確定性，可多個 next、可有 ε 轉移（等價於 DFA 但狀態數可能指數級）
PDA
Pushdown Automaton
FSM + 堆疊，能辨識 context-free 語言（如平衡括號）
Turing Machine
FSM + 無限 tape，理論計算上限，等價於所有可計算模型

九維屬性

哲學
計算能力階層
基本單位
State + Transition (+ Stack / Tape)
狀態位置
單一當前狀態（可能伴隨無限記憶體）
並行單位
無（純序列）
同步機制
無
形式化
★★★★★ 極強（Chomsky Hierarchy）
適合場景
正則表達式、編譯器 parser、理論計算研究
典型代表
regex 引擎、lex/flex、yacc/bison、ANTLR

優缺點

優點

  • 數學嚴謹，可判定性清楚
  • 能力階層明確
  • 理論研究基礎

缺點

  • 實務太底層
  • PDA 以上實作複雜
  • 無並行、無模組化

範例

平衡括號
DFA 辦不到無限深度 '(((...)))'，需要 PDA 的堆疊
規則語言
`^a*b$` 是 regex（DFA），`a^n b^n` 要 PDA，`a^n b^n c^n` 要 Turing

輸出位置

Moore vs Mealy ★★☆☆☆ 經典分類

輸出綁在狀態上？還是綁在轉移上？

FSM 經典兩變體：Moore 的輸出只看狀態；Mealy 的輸出看狀態+輸入。兩者表達能力等價，但建模風格完全不同。

左半 Moore：output 印在圓圈裡；右半 Mealy：output 印在箭頭上

成員

Moore Machine
output = f(state)，輸出綁在狀態上。直觀易 debug，狀態數通常較多
Mealy Machine
output = f(state, input)，輸出綁在 transition 上。狀態數較少，反應快

九維屬性

哲學
輸出=狀態函數 vs 輸出=狀態×輸入函數
基本單位
State + Transition + Output labels
狀態位置
單一當前狀態
並行單位
無
同步機制
時鐘邊緣（硬體）或事件循環（軟體）
形式化
強（可證明表達能力等價）
適合場景
硬體電路設計、Lexer tokenizer、協議狀態機
典型代表
Verilog/VHDL FSM、Digital circuit synthesis、Lexer generators

優缺點

優點

  • Moore：易 debug、易視覺化
  • Mealy：狀態數少、反應快一個時鐘週期

缺點

  • Moore：狀態可能膨脹
  • Mealy：輸出分布在 transition 上追蹤難

範例

紅綠燈 (Moore)
燈色只跟狀態有關 — 紅就是紅，不管輸入
販賣機 (Mealy)
按飲料鈕時出什麼，同時取決於「當前金額+按鈕」

擴展特性

EFSM / HSM / Statechart / UML ★★★★★ 工程最實用

解決 FSM 狀態爆炸 — 加條件、加階層、加並行

工程實務最該懂的一層。純 FSM 面對複雜系統會狀態爆炸。三法寶：Guard（條件）、Hierarchy（階層）、Orthogonal regions（並行區域）。

巢狀矩形 — EFSM 內虛線切割兩個並行 Region，同步發亮

成員

EFSM
Extended FSM
加 Guards（條件守門員）+ variables。Transition 不只看事件，還看變數。沒 Guard 的 FSM 幾乎是玩具
HSM
Hierarchical State Machine
狀態可巢狀。Superstate 的 transition 自動繼承給所有 substate（global transition 寫一次）
Statechart
Harel Statechart, 1987
HSM + orthogonal regions（並行狀態）+ history + entry/exit actions。工程 FSM 的終極型態
UML State Machine
UML 把 Harel Statechart 標準化，加 Protocol SM、Junction/Choice pseudostates

九維屬性

哲學
在 FSM 上堆疊條件 / 階層 / 並行三層能力
基本單位
Superstate + Substate + Region + Guard
狀態位置
階層樹 + 多 region 同時 active
並行單位
Orthogonal region（同時處於多個狀態）
同步機制
事件廣播到所有 active region
形式化
中強（比 FSM 複雜但仍可驗證）
適合場景
UI 框架、複雜 lifecycle、嵌入式系統、醫療設備
典型代表
XState (JS)、SCXML、python-statemachine、QP framework (C)、Stateflow (MATLAB)

優缺點

優點

  • 解決狀態爆炸（-90% 狀態數）
  • 模組化 subtree 可重用
  • 並行 region 表達多維狀態

缺點

  • 學習曲線陡（特別是 Statechart）
  • 工具支援不如 FSM 普及
  • 對簡單問題 overkill

範例

App 全域按 ESC 回主選單
純 FSM 要對每個子狀態寫 transition（10 狀態寫 10 條）。HSM 寫在 superstate 一次搞定
音訊播放器 orthogonal
同時有「播放 (Playing/Paused)」+「音量 (Muted/Unmuted)」+「網路 (Online/Offline)」— 三個 region 獨立

機率 / 時間

Markov, HMM, Timed Automaton ★★★★☆ ML / 實時系統

狀態不確定、看不見、或受時間約束

當系統非確定性、或要處理時間，FSM 無法表達。機率狀態機用分布取代確定轉移；HMM 從觀察推隱狀態；Timed Automaton 加時鐘處理實時約束。

左半 Markov：機率路徑擴散（粗路徑=高機率）；右半 Timed：時鐘 + 時間軸刻度

成員

Markov Chain
Transition 是機率分布 P(s'|s)。無記憶性（Markov property）
HMM
Hidden Markov Model
狀態不可見，只能從 emission 觀察推理。Viterbi/Forward-Backward 算法
MDP
Markov Decision Process
加 action + reward。強化學習理論基礎
POMDP
Partially Observable MDP
MDP + 部分可觀察。機器人決策、對話系統
Timed Automaton
FSM + clock 變數，transition 帶時間約束（如 'x < 5s'）
Hybrid Automaton
離散狀態 + 連續動態（微分方程），用於控制系統

九維屬性

哲學
不確定性、隱藏性、時間性是一等公民
基本單位
State + Transition(機率/時間) + Observation
狀態位置
機率分布 over states，或 state + clock variables
並行單位
無（多為單執行緒推理）
同步機制
時間步驟（離散 tick）或連續時間
形式化
★★★★★ 強（Viterbi、model checking）
適合場景
語音識別、NLP、PageRank、RL、實時控制、生物序列
典型代表
hmmlearn, Kaldi, Stan, UPPAAL (Timed Automaton verifier)

優缺點

優點

  • 表達不確定性
  • 可從觀察推隱狀態
  • 時間模型精確

缺點

  • Markov 假設限制（無長距依賴）
  • HMM 被 Transformer 超越
  • 實作成本高

範例

語音識別 (HMM)
音訊 → 音素序列。音素是隱狀態，MFCC 特徵是 observation
紅綠燈倒數 (Timed)
綠燈→黃燈條件：「x ≥ 30 秒」。clock 變數 x 在進入狀態時重置

並行 / 通訊

Petri Net, CSP, I/O Automaton ★★★★☆ 分散式核心

多件事並行 + 有同步點

FSM 是單執行緒模型，無法表達「多事同時做且要同步」。這層原生支援並行、同步、通訊。

Petri Net — token 從 Place 流向 Transition，湊齊才 fire，爆出 barrier 光效

成員

CFSM
Communicating FSM
多個 FSM 透過 channel 傳訊息。最簡單的並行 FSM 延伸
I/O Automaton
輸入/輸出明確分離，便於組合證明
Petri Net
Tokens 在 Places 間流動，Transition 消耗/產生 tokens。原生並行 + 同步 barrier + 可證 deadlock-free
π-calculus
進程代數，channel 名是一等公民，可動態傳遞
CSP
Communicating Sequential Processes
Tony Hoare 提出。Go 的 channel 受它啟發
CCS
Calculus of Communicating Systems
Milner 提出，π-calculus 前身

九維屬性

哲學
並行是一等公民，同步點可精確建模
基本單位
Petri: Places+Transitions+Tokens / 代數: Process+Channel
狀態位置
全域 token 分布 / 多進程組合狀態
並行單位
Transition 層級（多個可同時 fire）
同步機制
多輸入 transition 天然是 barrier（等所有 token 到才 fire）
形式化
★★★★★ 極強（可證 deadlock-free, liveness, boundedness）
適合場景
Workflow、BPMN、製造排程、通訊協議驗證、半導體製程
典型代表
YAWL, CPN Tools, UPPAAL, Camunda/Activiti, FDR (CSP)

優缺點

優點

  • 原生並行 + 同步
  • 數學可證 deadlock-free
  • workflow 建模直觀

缺點

  • 學習曲線陡
  • 狀態空間爆炸在大型系統
  • 工具偏學術

範例

訂單流程 (Petri)
「扣庫存」+「扣款」並行產生 tokens；「發貨」transition 需要兩個 token 都到才 fire（天然 barrier）
生產線 (Petri)
原料→機台→產品，token 計數表達有限資源

替代模型

BT / Rules / Actor / Dataflow ★★★★★ 實務主力

不是狀態機，但解類似問題

當狀態思維不是最佳抽象時，用其他範式：樹狀決策、規則匹配、訊息傳遞、資料流推動。每種在特定場景都勝過 FSM。

四象限：BT 樹狀 tick / Rules Engine 規則閃光 / Actor 訊息傳遞 / Dataflow 鏈式傳播

成員

Behavior Tree
優先級決策樹。tick-based pull 模型。Composite (Sequence/Selector/Parallel) + Decorator + Leaf。遊戲 AI 主流
Rules Engine
條件匹配觸發（if-then），無狀態概念，Rete 算法。適合合規/業務規則
Actor Model
Hewitt 1973
每個 actor 狀態私有，只透過訊息互動。Erlang 發揚光大，分散式系統基石
Dataflow / Reactive
Signal 變化自動傳播。spreadsheet、React、RxJS、Svelte runes 都是這類

九維屬性

哲學
不用狀態，用「樹/規則/訊息/資料流」作為主要抽象
基本單位
BT: Tree Node / Rules: if-then / Actor: Mailbox+Message / Dataflow: Signal+Derivation
狀態位置
BT: Blackboard / Actor: 每 actor 私有 / Dataflow: Signal 圖
並行單位
BT: Parallel node / Actor: 每 actor 獨立 / Dataflow: 自動並行
同步機制
BT: tick / Actor: message ack / Dataflow: subscribe/push
形式化
弱到中（Actor 少、Dataflow 有 FRP）
適合場景
BT: 遊戲 AI、機器人 / Rules: 合規 / Actor: 電信、高並發 / Dataflow: UI、spreadsheet
典型代表
BehaviorTree.CPP, py_trees, Unreal / Erlang, Akka, Orleans, Ray / Drools, CLIPS / React, RxJS, Svelte

優缺點

優點

  • 各自在主場完勝 FSM
  • 模組化、可重用
  • 更符合該領域直覺

缺點

  • 跨範式學習成本
  • 混用時介面複雜
  • 形式化不如 FSM 強

範例

遊戲 AI (BT)
Selector(逃跑/戰鬥/巡邏)，優先級由樹結構天然表達，加新行為不動其他節點
WhatsApp 50 工程師撐 9 億用戶 (Actor)
Erlang Actor Model + Supervisor 樹 + Let It Crash 哲學

實務混用場景

Hybrid Architectures — 五種經典組合 ★★★★★ 架構實戰

真實系統從不是單一模型 — 分層混用才強

實務中幾乎沒有純 FSM / 純 Actor 的系統。複雜系統按「外層管 WHAT、中層管 HOW、葉層管 WHEN」分層，每層用最適合的模型。

WHAT / HOW / WHEN 三層抽象 — 粒子逐層穿透後回傳

五種經典混用架構

🎮 遊戲 Boss 戰三層架構
FSM → Behavior Tree → FSM
Layer 1 外層 FSM：Boss 階段切換 — Phase1 → Phase2 → Phase3 由血量觸發
Layer 2 中層 BT：每階段內 AI 決策 — Selector(補血/召喚/範圍技/普攻)
Layer 3 葉 FSM：技能協議 — 蓄力 → 施法 → 冷卻的剛性流程
why：階段切換是離散單向→FSM；階段內選做啥→BT 優先級；技能是剛性協議→FSM
reps：Dark Souls boss、AAA 遊戲 AI
📦 分散式訂單系統
Petri Net → Actor Model → FSM
Layer 1 外層 Petri：整體 workflow 建模 — 下單/付款/出貨/完成的並行同步
Layer 2 中層 Actor：每張訂單一個 Actor，Payment/Shipping Actor 各司其職
Layer 3 葉 FSM：PaymentActor 內部 pending/paid/refunded 生命週期
why：Petri 建模可證 deadlock-free；Actor 天然並發容錯；FSM 管細節
reps：電商訂單中台、Shopify、Stripe 後端
🎙️ 語音助手 Pipeline
HMM → Statechart → BT
Layer 1 輸入層 HMM：語音識別 — MFCC 特徵推音素隱狀態
Layer 2 對話層 Statechart：多輪對話 lifecycle — Idle/Listening/Thinking/Speaking + orthogonal regions
Layer 3 決策層 BT：意圖路由 — Selector(查天氣/設鬧鐘/閒聊/fallback)
why：HMM 序列推理；Statechart 管對話狀態；BT 決定呼叫哪個技能
reps：Siri（早期）、Alexa、Kaldi 系統
🚗 自駕車即時系統
Petri Net → Timed Automaton → BT → Actor
Layer 1 管線層 Petri：感知→預測→規劃→控制 pipeline 的並行同步
Layer 2 安全層 Timed：ms 級實時約束 — 「剎車訊號 < 100ms 到位」可驗證
Layer 3 決策層 BT：駕駛行為 — Selector(緊急停車/變道/巡航/跟車)
Layer 4 通訊層 Actor：模組間訊息傳遞 — Sensor/Planner/Controller Actor
why：Petri 證 pipeline 無 deadlock；Timed 證實時上限；BT 決定動作；Actor 隔離故障
reps：Waymo、Tesla Autopilot、Apollo
⚡ 現代前端 Web App
Statechart → Dataflow → Actor
Layer 1 畫面層 Statechart：複雜 UI state — 表單多步驟、modal stack、auth flow
Layer 2 資料層 Dataflow：React/Svelte signal 自動傳播 derived state
Layer 3 背景層 Actor：Web Worker / Service Worker 做 offline sync、heavy compute
why：Statechart 管 UI 複雜度；Dataflow 管資料響應；Actor 管背景計算隔離
reps：XState + React、Linear、Figma
分層心法
外層 管 WHAT（做什麼）
中層 管 HOW（怎麼選）
葉層 管 WHEN（如何執行）
每層選最適合的模型，組合比單選強十倍。
