I/O Automaton Input/Output Automaton

輸入 / 輸出明確分離 — 便於組合證明

哲學

Nancy Lynch 在《Distributed Algorithms》中提出。核心規則只有兩條：input actions 不可拒絕（環境主導），output actions 自主發生（automaton 主導）。這個非對稱設計讓多個 I/O Automaton 組合後仍是 I/O Automaton，可組合性嚴格成立，適合分散式算法的形式化證明。

九維屬性

哲學input / output 不對稱，可組合性嚴格保證
基本單位States + Input actions + Output actions + Internal actions + Transition relation
狀態位置單一 automaton 的局部狀態
並行單位多 automaton 合成（product automaton）
同步機制Action matching（output of A = input of B）
形式化★★★★★ 極強（Lynch-Tuttle 理論，分散式算法標準工具）
適合場景共識算法形式化、分散式互斥、Byzantine agreement 證明
典型代表IOA Language（MIT）、Nancy Lynch《Distributed Algorithms》、TLA+（相關思想）

具體例子

例 1：Paxos 形式化
Proposer Automaton
  input:  client_request(v)       -- 不可拒絕
  output: prepare(n)              -- 自主發出
  output: accept(n, v)            -- 自主發出
  internal: choose_value          -- 內部決策

Acceptor Automaton
  input:  prepare(n), accept(n,v) -- 不可拒絕
  output: promise(n), accepted(n,v)

兩個 automaton 的 output/input 對接後，合成 automaton 保留 I/O Automaton 結構，可直接證明 safety / liveness。

例 2：分散式互斥

每個節點是一個 I/O Automaton，輸入是其他節點的 release 訊號，輸出是 request 進入 CS。合成後可證明「任意時刻至多一個節點在 CS」。

優缺點

✓ 優點
  • 可組合性強：多 automata 合成仍是 I/O Automaton
  • 形式化證明嚴謹，Safety / Liveness 分析有完整理論
  • 適合分散式系統的不變式推導
✗ 缺點
  • 實務工程很少直接使用，學術味重
  • 工具鏈小眾（IOA Language 維護度低）
  • 學習曲線陡：需要紮實的形式化方法背景

代表實作

  • IOA Language（MIT）：Lynch 團隊開發，直接對應 I/O Automaton 語義
  • Nancy Lynch《Distributed Algorithms》：權威教材，所有主流分散式算法的形式化基礎
  • TLA+（Lamport）：相近思想，更廣泛的工業採用（Amazon、Microsoft）
  • TIOA Toolkit：Timed I/O Automaton 的分析工具
