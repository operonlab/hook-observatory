Ⅳ 機率 / 時間

Timed Automaton Timed Automaton / Real-Time Automaton

FSM + clock 變數 — transition 帶時間約束，可形式驗證時間 bound

哲學

實時系統需要「毫秒級可驗證」的保證。Timed Automaton 把連續時間 clock 納入 FSM，讓 transition guard 可以寫出 x ≤ 5 這類時間約束。系統從設計期就能用 model checker（UPPAAL）證明「煞車響應必在 100ms 內」，而不是靠測試猜測。

九維屬性

哲學FSM 上的 clock 變數讓時間約束成為一等公民，支援形式驗證
基本單位States + Transitions + Clocks（連續實數）+ Clock constraints（guards）+ Clock resets
狀態位置（離散狀態，clock 向量）的組合 — 稱為 configuration
並行單位多個 Timed Automaton 可並行組合（Network of TA）
同步機制Channel 同步（UPPAAL 的 chan! / chan? 握手）
形式化★★★★★ 極強（可達性分析決定性、TCTL 時間邏輯可驗證、region/zone 圖有限）
適合場景安全關鍵系統（煞車/起落架）、通訊協議超時、紅綠燈控制、嵌入式即時 OS
典型代表UPPAAL、Kronos、Rabbit、TINA

具體例子

例 1：交通號誌控制器
狀態：{Green, Yellow, Red}
Clock：x（號誌計時器）

Green → Yellow：guard x ≥ 30, reset x := 0
Yellow → Red：guard x ≥ 5,  reset x := 0
Red   → Green：guard x ≥ 25, reset x := 0

Invariant：Green: x ≤ 35（不能超時停在綠燈）

UPPAAL 可自動驗證「從 Red 到 Green 最多等 30 秒」這類時間安全性質。

例 2：ABS 煞車系統

感測器偵測到打滑後，煞車泵必須在 100ms 內響應。用 Timed Automaton 建模整條控制鏈，UPPAAL 驗證最壞情況 latency 不超過 deadline。車用安全標準 ISO 26262 要求此類形式驗證。

例 3：TCP 重傳超時（Timeout Protocol）

兩個並行 Timed Automaton：Sender（等待 ACK，clock x）與 Receiver（處理 + 回傳）。Clock guard x ≥ TIMEOUT 觸發重傳 transition。UPPAAL 驗證在封包最大延遲假設下，重傳邏輯不會死鎖。

優缺點

✓ 優點
  • 時間可驗證：UPPAAL 可自動證明或反例展示時間安全性質
  • 形式化成熟：可達性分析決定性，zone 抽象讓狀態空間有限
  • 可生成測試案例：從反例路徑自動生成測試輸入
✗ 缺點
  • Clock 爆炸：多個 clock 的 zone graph 指數增長
  • 規格難撰寫：TCTL 時間邏輯對工程師門檻高
  • 實務採用率低：多數工程師改用測試代替驗證，學術界為主

代表實作

  • UPPAAL：最強 Timed Automaton model checker（GUI + DBM zone 引擎）
  • Kronos：學術 TCTL 驗證器，支援完整時間邏輯語法
  • Rabbit：Timed Automaton → C 程式碼生成器
  • TINA：Time Petri Net + Timed Automaton 複合工具
