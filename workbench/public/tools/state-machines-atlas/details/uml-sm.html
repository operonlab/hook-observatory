Ⅲ 擴展特性

UML State Machine UML 2.x Behavioral / Protocol State Machine

工業標準化的 Statechart — 加 Protocol SM 與豐富 pseudostates

哲學

把 Harel Statechart 標準化進 UML 2.x 規範，並加入 Protocol State Machine 這個實用擴展——用來限制對物件的合法呼叫順序，而非只描述內部行為。豐富的 pseudostate（Junction、Choice、Fork、Join、History）讓複雜分支和並行都有標準畫法。代價是規範本身龐大，工具鏈昂貴。

九維屬性

哲學標準化 Statechart + Protocol SM（合法呼叫順序約束）
基本單位State / Transition / Entry·Exit·Do / Junction / Choice / Fork / Join / Shallow History / Deep History / Final
狀態位置層次路徑 + 並行 region（同 Statechart）

形式化★★★★★ 極強（ISO/IEC 19505，OMG UML 2.x spec）
適合場景醫療設備 lifecycle、銀行帳戶 API 順序約束、FDA/CE 認證系統建模
典型代表IBM Rational Rhapsody、Enterprise Architect、PlantUML、Papyrus、Qt State Machine

具體例子

例 1：銀行帳戶 Protocol State Machine
Protocol SM — 合法呼叫順序：
  PENDING ──open()──▶ ACTIVE
  ACTIVE  ──freeze()──▶ FROZEN
  FROZEN  ──unfreeze()──▶ ACTIVE
  ACTIVE  ──close()──▶ CLOSED

非法：CLOSED 狀態呼叫 deposit() → 編譯期或執行期報錯

Protocol SM 不描述內部實作，只規定哪些操作在哪個狀態下合法。用於 API 設計的契約驗證。

例 2：線上購物訂單（Junction 分支）

PAID 狀態後，透過 Junction pseudostate 依 stock_available 條件分支：庫存充足走 SHIPPED，缺貨走 BACKORDERED。Junction 是靜態分支（transition 前計算），Choice 是動態分支（transition 後計算）。

例 3：醫療設備嚴格 lifecycle

ISO 13485 認證要求設備狀態圖符合 UML SM 規範。審核者直接看 UML SM 圖確認 lifecycle 合規，工具自動生成文件和程式碼，確保實作與規格一致。

優缺點

✓ 優點
  • 工業標準：FDA/CE/ISO 認證常要求 UML SM 建模文件
  • 豐富 pseudostate：Junction、Choice、Fork、Join 應對各種分支並行場景
  • 工具鏈成熟：code gen、simulation、model checking 一條龍
✗ 缺點
  • UML 被批「overengineered」：spec 500+ 頁，大多數場景用不到
  • 工具昂貴：Rational Rhapsody、Enterprise Architect 授權費高
  • 手繪耗時：pseudostate 多，圖形複雜，維護成本高

代表實作

  • 商業工具：IBM Rational Rhapsody（C++/Java codegen）、Enterprise Architect（Sparx）
  • 開源：Papyrus（Eclipse EMF based）、PlantUML（@startuml 文字定義）
  • C++/Qt：Qt State Machine Framework（QStateMachine）
  • 標準參考：OMG UML 2.5.1 spec（www.omg.org/spec/UML）
