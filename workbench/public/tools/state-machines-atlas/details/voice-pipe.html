Ⅷ 實務混用場景

🎙️ 語音助手 Pipeline

HMM (ASR) → Statechart (對話) → BT (意圖) — AI 語音 pipeline 經典組合

哲學

語音助手的挑戰分三層：信號層（音訊如何轉成文字）、對話層（多輪上下文如何管理）、決策層（用戶意圖如何路由到技能）。三層性質截然不同。

HMM 擅長序列概率推理——音素之間的隱狀態轉移，天生是統計序列問題。Statechart 管對話的複雜狀態——有嵌套（Listening 內有 WakeWord / Content 子狀態）、有並行（auth region ∥ context region ∥ UI region）。BT 管意圖路由——天氣優先、鬧鐘次之、fallback 保底，帶優先級的選擇邏輯。

分層架構

Layer 1 (輸入層)HMM / 語音識別 / MFCC 特徵 → 音素隱狀態序列 → 單詞
Layer 2 (對話層)Statechart / 多輪對話 lifecycle / Idle / Listening / Thinking / Speaking + orthogonal regions (auth ∥ context ∥ UI)
Layer 3 (決策層)Behavior Tree / 意圖路由 / Selector(查天氣 / 設鬧鐘 / 播音樂 / 閒聊 / fallback)
資料流音訊 → HMM 解碼 → NLU 抽意圖 → Statechart 更新對話狀態 → BT 選對應技能 → 回應
核心優勢序列概率推理（HMM）+ 對話狀態完整建模（Statechart）+ 技能可無限擴展（BT）

架構示意

// Layer 1: HMM (Kaldi / Whisper)
audio → MFCC features → Viterbi decode → ["what", "is", "the", "weather"]

// Layer 2: Statechart (Rasa / Dialogflow CX)
state: Idle
event: WakeWord → Listening
event: Transcript("what is the weather") → Thinking
  [orthogonal] context.update(intent="weather", slot.city="Taipei")
event: ResponseReady → Speaking → Idle

// Layer 3: BT (intent router)
Selector
  ├─ Sequence [intent=="weather"] → WeatherSkill.run(city)
  ├─ Sequence [intent=="alarm"]   → AlarmSkill.run(time)
  ├─ Sequence [intent=="music"]   → MusicSkill.run(query)
  └─ Action → ChitchatFallback

具體例子

例 1：Siri 早期架構（Apple Intelligence 前）

Nuance HMM-based ASR → 對話管理器（Statechart，管 follow-up 問題的狀態）→ domain router（意圖 BT，決定啟動 Maps / Calendar / Weather domain）。domain 擴展只需在 BT 加葉節點，不動 Statechart。

例 2：Alexa Skills Kit

Lex ASR → Alexa 對話狀態機（session attributes 構成隱式 Statechart）→ intent handler 樹（優先級 BT，InvokeSkill → FallbackIntent）。第三方 Skill 只需實作 BT 葉節點，是 BT 擴展性的最佳示範。

例 3：現代 LLM-based 語音助手

HMM 已被 Whisper / Conformer（end-to-end DL）取代，ASR 準確率大幅提升；但 Statechart 管 session 狀態（streaming 中 / function calling 等待）+ BT 路由工具調用（search vs code vs browse）的架構仍保留，因為這兩層的問題性質沒有變。

何時採用

語音助手需要支援 5 個以上意圖、且有多輪對話（slot filling、澄清問題、context carry-over）時，這個三層組合最合適。若只有單輪 command，直接用 intent classifier + FSM 即可；若完全由 LLM 管對話，Statechart 可退化成簡單的 session state dict。

優缺點

✓ 優點
  • 模組化——每層可獨立升級（HMM 換 Whisper，不動 Statechart）
  • 可解釋性強——Statechart 視覺化，BT tick log 可追
  • 成熟工具鏈——Kaldi / Rasa / Dialogflow 都按此分層設計
  • 技能擴展容易——加新意圖只需在 BT 加葉節點
✗ 缺點
  • HMM 在噪音環境表現被 end-to-end DL 全面超越
  • Statechart 設計複雜——orthogonal regions 的 history states 容易出錯
  • 跨層通訊介面需要標準化（NLU 輸出格式 → Statechart event）
  • 多語言支援需要每層分別處理，難以統一
