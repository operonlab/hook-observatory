Ⅲ 擴展特性

HSM Hierarchical State Machine

狀態可巢狀 — superstate 的 transition 自動繼承給所有 substate

哲學

「共用 transition」的優雅解。傳統 FSM 裡，若十個 state 都要對 ESC 鍵返回主選單，就要畫十條重複的箭頭。HSM 把這十個 state 放進一個 superstate，只在 superstate 層寫一條 ESC transition，所有 substate 自動繼承。巢狀帶來的是「行為的繼承」，而非資料的繼承。

九維屬性

哲學Superstate / Substate 巢狀，transition 向上繼承
基本單位Superstate + Substate + Entry/Exit actions + Nested transitions
狀態位置層次路徑（superstate → substate → sub-substate）
並行單位無（純層次，不並行）
同步機制Entry/Exit actions（進出狀態時執行）
形式化★★★★☆ 強（Harel Statecharts 子集，有明確語義）
適合場景UI 導航、遊戲角色 lifecycle、嵌入式裝置控制、家電韌體
典型代表Miro Samek QP framework、XState nested states、Statecharts（Harel）

具體例子

例 1：App 全域返回主選單
Superstate: APP_ACTIVE
  ├── Substate: HOME
  ├── Substate: SETTINGS
  │     ├── SETTINGS_GENERAL
  │     └── SETTINGS_ACCOUNT
  └── Substate: PROFILE

APP_ACTIVE ──ESC──▶ MAIN_MENU   ← 所有 substate 自動繼承

無論在 SETTINGS_ACCOUNT 還是 PROFILE，按 ESC 都觸發同一條 transition，不需重複畫箭頭。

例 2：遊戲角色生命週期

Superstate ALIVE 包含 Idle / Walk / Run / Attack 四個 substate。on_death transition 只寫在 ALIVE 上，所有動作狀態都能觸發死亡。Dead 是另一個同層的 state。

例 3：電視遙控器

Superstate TV_ON 含 Channel / Volume / Menu 三個 substate。Power 按鈕的 transition 只寫一次在 TV_ON 上，關機邏輯由 Exit action 統一處理。

優缺點

✓ 優點
  • 大幅減少重複 transition：共用行為提升到 superstate
  • 符合直覺：「這一群 state 都是 Alive」的語義自然對應
  • Entry/Exit actions 集中管理進出副作用
✗ 缺點
  • 巢狀過深追蹤困難：需要維護 state stack
  • UML 圖巢狀後難閱讀，超過三層就混亂
  • 實作需要 state stack 支援，純 switch/case 無法勝任

代表實作

  • C/C++（嵌入式）：Miro Samek QP framework（Quantum Platform，書《Practical UML Statecharts in C/C++》）
  • JavaScript：XState v5 states 巢狀（最成熟的 HSM 實作）
  • Python：python-statemachine（部分 HSM 支援）
  • 理論基礎：David Harel 1987 論文《Statecharts: A Visual Formalism for Complex Systems》
