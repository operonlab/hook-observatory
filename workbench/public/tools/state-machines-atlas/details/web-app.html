Ⅷ 實務混用場景

⚡ 現代前端 Web App

Statechart (UI) + Dataflow (reactivity) + Actor (background) — 前端三層現代架構

哲學

前端複雜度在三個維度爆炸：UI 狀態（多步表單、modal stack、auth flow 互相疊加）、資料響應（衍生值自動更新、跨元件同步）、背景任務（offline sync、heavy compute 不能卡主執行緒）。三個維度用同一種抽象都會失控。

Statechart 管 UI 的複雜狀態——有嵌套（Authenticated 內有 Browsing / Editing 子狀態）、有 history（離開再回來恢復上次狀態）、有 guard（未儲存時離開要 confirm）。Dataflow 管資料響應——derived store 自動傳播，不需手動觸發。Actor 管背景任務——Web Worker 做 IndexedDB sync，主執行緒不感知延遲。

分層架構

Layer 1 (畫面層)Statechart / 複雜 UI state / 多步表單、modal stack、auth flow、routing state 嵌套管理
Layer 2 (資料層)Dataflow / Reactive / Signal 自動傳播 / React hooks、Svelte runes、Zustand derived store 自動更新
Layer 3 (背景層)Actor Model / Web Worker / Service Worker 做 offline sync、heavy compute、IndexedDB queue
資料流User interaction → Statechart 轉換 state → Dataflow 更新 derived → Render → 背景 Actor 處理同步 / heavy task
核心優勢UI 複雜度可管（Statechart）+ 資料響應自動（Dataflow）+ 主執行緒不阻塞（Actor）

架構示意

// Layer 1: Statechart (XState v5)
const machine = createMachine({
  initial: 'idle',
  states: {
    idle:    { on: { SUBMIT: 'submitting' } },
    submitting: { on: { SUCCESS: 'done', ERROR: 'error' } },
    done:    { type: 'final' }
  }
})

// Layer 2: Dataflow (Svelte 5 Runes)
const total = $derived(items.reduce((s, i) => s + i.price, 0))
const filtered = $derived(items.filter(i => i.status === activeFilter))
// total / filtered 自動追蹤 items 依賴

// Layer 3: Actor (Web Worker)
// main.ts
const worker = new Worker('./sync.worker.ts')
worker.postMessage({ type: 'SYNC', payload: pendingOps })
worker.onmessage = ({ data }) => store.set('syncStatus', data)
// sync.worker.ts: 全部 IndexedDB / fetch 在 worker 內，不阻塞 UI

具體例子

例 1：XState + React

XState 管頁面 flow（登入→MFA→首頁 Statechart）；React hooks / Zustand 管資料 Dataflow；Web Worker 做 CSV 匯出等 heavy compute。XState 官方 React 範例即此架構，有完整 devtools 視覺化。

例 2：Linear.app

Statechart 管 Command Palette / 多步 modal 的 UI 狀態；reactive store 管 issue list 的 derived filter；Service Worker Actor 做 offline-first 同步，確保斷線後操作不丟失。Linear 公開分享了這套架構設計。

例 3：Figma / Notion

Figma 把 canvas rendering 放進 Web Worker Actor（WASM + OffscreenCanvas），主執行緒只管 UI Statechart，60fps 不卡；Notion 把 CRDT 協同計算放進 Worker，reactive store 只訂閱最終結果，渲染層完全不感知計算複雜度。

何時採用

前端有複雜多步流程（3 個以上步驟的表單、modal 疊加、offline 需求）時，三層組合最合算。若只是 CRUD 頁面，直接用 React state + fetch 就夠；若有 realtime 協作，Layer 3 Actor 換成 CRDT + WebSocket worker 即可，其餘不動。

優缺點

✓ 優點
  • UI 複雜度分層——Statechart 可視覺化，設計師能讀懂狀態流
  • 可測試性強——Statechart 可 unit test 每個轉移，無需渲染 DOM
  • 主執行緒輕量——heavy task 在 Worker Actor，UI 60fps 不卡頓
  • 現代 PWA 架構——Service Worker Actor 天然支援 offline-first
✗ 缺點
  • 三套概念學習成本高，新人需要同時理解 Statechart + Signals + Actor
  • 工具整合複雜——XState + React + Web Worker 的 devtools 各自獨立
  • Worker 通訊只能序列化資料，不能傳遞函數或 DOM 引用
  • bundle size 增加——XState v5 約 15KB gzip，需評估是否值得

代表實作

  • XState v5 + React：官方推薦的 Statechart + hooks 組合，有完整 inspector devtools
  • Svelte 5 + Runes：內建 Statechart（$state）+ Dataflow（$derived），無需外部庫
  • Replicache / Jazz Tools：封裝 Worker Actor + CRDT Dataflow，offline-first 開箱即用
  • InstantDB：Dataflow + Actor 同步，類 Firebase 但完全開源
  • Linear tech stack：公開分享了 Statechart + offline Actor 架構，業界最佳參考之一
