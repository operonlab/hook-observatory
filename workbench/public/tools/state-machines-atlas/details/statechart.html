Ⅲ 擴展特性

Statechart Harel Statechart

HSM + orthogonal regions + history + entry/exit — 工程 FSM 的終極型態

哲學

David Harel 在 1987 年提出，用三個法寶解決 FSM 的所有痛點：階層（HSM）消除重複 transition、並行 region（orthogonal）讓多個維度同時活躍、history pseudostate 讓系統記住上次離開時的位置。三者合一，Statechart 能以接近線性的狀態數表達指數級的系統複雜度。

九維屬性

哲學HSM + 並行 region + history：階層、並行、記憶三位一體
基本單位Hierarchical states + Orthogonal regions（∥）+ History pseudostates + Actions
狀態位置每個 region 各自有當前狀態（多個活躍狀態同時存在）
並行單位Orthogonal regions（同一 superstate 內多個獨立 region）
同步機制Entry/Exit actions、internal transitions、broadcast events
形式化★★★★★ 極強（UML State Machine 的理論基礎，有 SCXML W3C 標準）
適合場景音訊播放器、手機 OS、UI 框架、任何有多個獨立維度的複雜系統
典型代表XState（JS）、SCXML（W3C）、python-statemachine、Yakindu

具體例子

例 1：音訊播放器（三個並行 region）
Superstate: PLAYER_ACTIVE
  ∥ Region PLAYBACK:  STOPPED | PLAYING | PAUSED
  ∥ Region VOLUME:    NORMAL  | MUTED
  ∥ Region NETWORK:   ONLINE  | BUFFERING | OFFLINE

三個維度完全獨立：可以同時「PLAYING + MUTED + BUFFERING」。純 FSM 需要 3×2×3 = 18 個 state，Statechart 只需 3+2+3 = 8 個。

例 2：手機並行運作

通話 region、音樂 region、WiFi region 各自獨立運作。接到電話時通話 region 進入 RINGING，不影響 WiFi region 的 CONNECTED 狀態。

例 3：History Pseudostate

使用者在 SETTINGS 的深層子頁面（SETTINGS_ACCOUNT_SECURITY），按 Home 鍵暫離。下次進 SETTINGS 時，Deep History pseudostate 直接恢復到 SETTINGS_ACCOUNT_SECURITY，而非 SETTINGS 的初始 substate。

優缺點

✓ 優點
  • 表達力接近 UML SM：一張圖搞定複雜 lifecycle
  • Orthogonal region 解決「同時多狀態」的本質問題
  • History 讓 UX 流程更自然（記住上次位置）
✗ 缺點
  • 學習曲線陡：region、history、pseudostate 概念多
  • 需要專門函式庫支援，自己實作很容易出錯
  • debug 工具不如簡單 FSM 普及，視覺化難

代表實作

  • JavaScript：XState v5（最成熟的 Statechart 實作，有 Stately Studio 視覺化）
  • 標準：SCXML（W3C State Chart XML，XML 描述 Statechart）
  • Python：python-statemachine（支援 HSM，partial Statechart）
  • IDE 工具：Yakindu Statechart Tools（Eclipse，含 codegen）
