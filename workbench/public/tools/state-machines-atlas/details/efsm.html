Ⅲ 擴展特性

EFSM Extended Finite State Machine

FSM + Guards + Variables — transition 不只看事件，還看變數值

哲學

FSM 的最大痛點是狀態爆炸：每個細微差異都要拆成獨立 state。EFSM 把「重複的差異」吸收進變數，用 guard 條件守住 transition 入口。結果是：狀態圖保持精簡，業務邏輯透過變數和條件表達，而不是靠無數個幾乎相同的 state。

九維屬性

哲學FSM + 變數 + guard 條件，避免狀態爆炸
基本單位State + Transition + Guards（布林條件）+ Variables（整數、計數器等）
狀態位置單一當前狀態 + 一組變數值
並行單位無（純序列）
同步機制無（單執行緒邏輯）
形式化★★★☆☆ 中（形式化工具較少，狀態×變數空間難完備驗證）
適合場景登入流程、計數器、電梯、通訊協議 session 管理
典型代表python-statemachine（Guards）、XState guards + actions、SDL 建模語言

具體例子

例 1：登入重試限制
State: IDLE, LOCKED
Variable: retry_count = 0

IDLE ──login_fail [retry_count < 3] / retry_count++──▶ IDLE
IDLE ──login_fail [retry_count >= 3]──▶ LOCKED
IDLE ──login_ok──▶ AUTHENTICATED

沒有 EFSM 的話，需要 IDLE_1 / IDLE_2 / IDLE_3 / LOCKED 四個狀態。有了 guard，一個 IDLE 加一個 retry_count 變數搞定。

例 2：電梯

State 只有 IDLE / MOVING_UP / MOVING_DOWN / DOOR_OPEN。floor 變數記錄樓層，guard [floor == target] 決定是否停靠，省下為每一層建 state 的麻煩。

例 3：TCP 協議

sequence number、window size 都是變數；guard 條件篩選哪些 segment 才觸發 ACK transition。純 FSM 無法表達這層語義。

優缺點

✓ 優點
  • 狀態數大幅下降：差異用變數吸收，不用拆 state
  • Guard 讓條件聲明式：transition 本身清楚說明「何時觸發」
  • 靈活：variables 可以是計數器、enum、字串等任意型態
✗ 缺點
  • 形式化工具支援少：狀態×變數空間太大，難自動驗證
  • 測試案例難完備：變數值域造成無限多組合
  • guard 衝突難發現：多個 guard 可能同時為真，行為不確定

代表實作

  • Python：python-statemachine（條件函數當 guard）
  • JavaScript：XState guards + actions（v5 完整支援）
  • 通訊建模：SDL（Specification and Description Language，ITU-T 標準）
  • 測試：GraphWalker（基於 EFSM 模型自動生成測試路徑）
