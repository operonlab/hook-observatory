Ⅵ 替代模型

BT Behavior Tree

優先級決策樹 — tick-based pull 模型

哲學

BT 不追蹤「我在哪個狀態」，而是每 tick 從 root 重新決策「現在該做什麼」。這是 push（事件驅動）的對立面：pull 模型主動掃描全樹，依優先級回傳 SUCCESS / FAILURE / RUNNING。加行為 = 插 subtree，不動其他節點。

九維屬性

哲學每 tick pull 決策，優先級由樹結構隱性表達
基本單位Composite（Sequence / Selector / Parallel）+ Decorator + Leaf（Action / Condition）+ Blackboard
狀態位置Blackboard（共享記憶體）+ 節點內部 RUNNING 狀態
並行單位Parallel 節點（邏輯並行，不一定執行緒並行）
同步機制Blackboard 讀寫（無鎖，單執行緒 tick）
形式化★★★☆☆ 有語意定義，實作差異大，無統一形式規範
適合場景遊戲 AI、機器人控制、無人機任務規劃、NPC 行為設計
典型代表BehaviorTree.CPP、py_trees、Unreal Engine BT、NodeCanvas（Unity）

具體例子

例 1：遊戲 NPC 行為（Selector 優先級）
Selector（有一個 SUCCESS 就停）
├─ Sequence「逃跑」
│   ├─ Condition: HP < 20%
│   └─ Action: FleeToBase
├─ Sequence「戰鬥」
│   ├─ Condition: EnemyInRange
│   └─ Action: Attack
└─ Action: Patrol（預設行為）

優先級直接體現在樹結構中：先嘗試逃跑，再考慮戰鬥，最後巡邏。不需要任何 if/else 或 transition 定義。

例 2：機器人取物任務（Sequence 串行）
Sequence（任一 FAILURE 即中止）
├─ Action: MoveToTarget
├─ Action: PickUpObject
└─ Action: ReturnToBase

Sequence 像程式的 AND：前一步成功才執行下一步。任何 Action 回傳 FAILURE，整條 Sequence 立即中止。

例 3：無人機任務（Parallel 同時監控）
Parallel（兩個子樹同時 tick）
├─ Sequence「電量監控」
│   ├─ Condition: Battery < 15%
│   └─ Action: ReturnToCharge
└─ Sequence「任務執行」
    ├─ Action: ScanArea
    └─ Action: TransmitData

Parallel 讓電量監控和任務執行同步運作。電量不足時，監控分支觸發返航，任務分支被中斷。

優缺點

✓ 優點
  • 模組化強：subtree 可重用、跨角色共享
  • 優先級表達直覺：樹的左到右、上到下即優先序
  • 加行為不改現有節點：插 subtree，其他分支不受影響
  • tick 模型天生容錯：每幀重新決策，避免卡死在錯誤狀態
✗ 缺點
  • 每 tick 掃全樹：100+ 節點需手動優化（event-driven BT 或 lazy eval）
  • RUNNING 狀態管理複雜：中斷與恢復的語意因框架而異
  • debug 工具相對少：沒有 FSM 那種可視化轉移歷史
  • 全域狀態需 Blackboard 協調：設計不當容易產生隱性耦合

代表實作

  • C++：BehaviorTree.CPP（最主流，ROS 2 官方採用，支援 Groot 視覺化）
  • Python / ROS：py_trees（Python 原生，ROS 2 整合良好）
  • Unreal Engine：內建 Behavior Tree（UE AI Controller 官方方案）
  • Unity：NodeCanvas、Behavior Designer（Asset Store 主流框架）
  • Rust：bonsai-bt（輕量型 Rust 實作）
