🌲 行為樹教學

從小精靈吃豆子到約會決策 — 用生活例子徹底搞懂決策樹、行為樹、以及兩者的轉換陷阱

SECTION 01

什麼是行為樹？

想像小精靈是一個機器人小孩，他不會自己思考，只會照一張超大的待辦清單做事。這張清單畫成樹的形狀，就叫做「行為樹 (Behavior Tree)」。

每秒鐘（每個 tick），小精靈會從清單最上面看起，照順序問自己：「這件事我能做嗎？」能做就做，不能做就跳到下一個。

行為樹 vs 決策樹

🌳 決策樹 = 問卷調查

像做一份「你是哪種動物」的測驗：翻到結論頁就結束，不會再重做。

  • 一次性回答「這是什麼」
  • 葉子 = 結論（鴨子、蘋果）
  • 沒有「進行中」的概念

🤖 行為樹 = 每秒重啟的大腦

小精靈每秒都從頭想一次「我現在該幹嘛」。

  • 持續控制「該做什麼」
  • 葉子 = 動作（追鬼、吃豆子）
  • 動作有三種狀態：成功 / 失敗 / 進行中
SECTION 02

兩個核心符號

符號 1：? Selector（選擇器）

意思：「A 不行換 B，B 不行換 C，找一個能做的」

生活比喻：放學想吃點心
  • 先看有沒有餅乾？→ 沒有
  • 再看有沒有糖果？→ 沒有
  • 最後看有沒有水果？→ 有！吃水果 ✓

這就是 Selector：一個一個試，找到一個可以就停。

符號 2：-> Sequence（序列）

意思：「每一步都要做對，少一步就失敗」

生活比喻：泡泡麵
  • 燒開水 ✓
  • 把麵放進去 ✓
  • 等三分鐘 ✓
  • 撈起來吃 ✓

少一步（沒燒水就放麵）就泡不成。這就是 Sequence：每一步都要成功，整件事才算成功。

顏色慣例

顏色代表說明
黃色Condition條件判斷（問一個問題，回傳是/否）
綠色Action實際動作（追、逃、吃）
灰色Control控制節點（? / ->）

SECTION 04

決策樹 → 行為樹：四步驟

Step 1：把「問題」拆成黃色 Condition 葉子

決策樹裡「鬼靠近嗎？」是個分叉點。轉成 BT 時，它變成獨立的黃色方塊，回傳 Success（是）或 Failure（否）。

Step 2：每條路重寫成 Sequence ->

原本的路「鬼靠近 → 鬼害怕 → 追」變成 -> [鬼靠近?, 鬼害怕?, 追]，用 Sequence 把條件 + 動作串起來。

Step 3：多條路用 Selector ? 包起來

? Selector
├── -> [鬼靠近?, 鬼害怕?, 追]
├── -> [鬼靠近?, 逃]
└── -> [吃豆子]

Step 4：加預設動作當兜底

? Selector
├── -> [鬼害怕?, 追]
├── -> [鬼靠近?, 有藥丸?, 吃藥丸]
├── -> [鬼靠近?, 逃]
└── 吃豆子             ← 什麼條件都沒滿足時的預設
⚠ 講師在這裡翻車

講師做了「形狀轉換」但沒做「思維轉換」—— 把決策樹的 if-else 直接套上 ? -> 符號，沒重新設計優先順序、也忘了加預設動作。行為樹的核心是「優先順序 + 預設動作 + 每 tick 重跑」，這三件事決策樹裡都沒有，硬轉一定出漏洞。

🐛 為什麼講師版是錯的
  1. 語義重複：? 的 fallback 語義本來就是「試下一個」，講師又用巢狀 ? 再模擬一次 → 做白工
  2. 決策樹腦袋：講師還在用 if-else 思維畫 BT，沒換腦袋
  3. 擴充災難：加第四個女生 D 要再包一層，十個女生就十層
  4. 視覺誤導：樹越深，學生越以為邏輯很複雜，但其實超簡單
SECTION 06

這是瑞典皇家理工學院（KTH）課程 BT intro part 3 的例子，出自 Petter Ögren 教授（與 Michele Colledanchise 合著 BT 教科書《Behavior Trees in Robotics and AI》）。影片原標題是 "Creating Behavior Trees using the Decision Tree design"，介紹的是書中正式列出的 BT 設計方法之一：把既有的決策樹直接映射成行為樹。這個方法能快速把現成的 if-else 邏輯搬進 BT，但也會把 DT 的深度巢狀原封不動帶進來——這正是本節要討論的陷阱。

⚠ 這不是「故意做壞」，是方法本身的邊界

Decision Tree design 是 Ögren 書中並列介紹的 BT 設計方法之一（Part 2 是 Backward Chaining、Part 3 是 Decision Tree design），並非反面教材。Ögren 真正明確標示為反例的影片會直接寫在標題裡——例如 Part 5B："Why Memory Nodes is a bad idea in Behavior Trees"。

問題不在「Ögren 故意做壞」，而在這種方法的邊界：直接把 DT 套上 ? / -> 符號，會把 DT 的深度巢狀一併複製過來。如果下游的講師只抓 Part 3 的機械轉換例子當成終點，就會產出這種埋 5-6 層的戰士 BT。實務上要再搭配書中另一主軸——扁平、優先級、預設動作——才能把 DT 風格的 BT 收斂到可維護的結構。

SECTION 07

什麼時候階層該深、什麼時候該淺？

黃金原則

每多一層要能回答：「這層表達了什麼組合邏輯 (AND/OR/NOT)？」

答不出來，這層就是多餘的。

該不該保留這層的判斷法

這層拿掉後判決行動
語義不變❌ 冗餘扁平化
語義會變✅ 有意義保留

合理的多層範例

3 層結構：每層都有組合語義 — 約會 OR 備案 / 選對象 AND 準備 / 條件 + 動作

SECTION 08

一句話懶人包

💡 記住這兩句

決策樹回答「這個是什麼」，行為樹回答「我現在該做什麼」。

行為樹的精髓是「用符號省掉結構」 — 符號已經表達的語義，別再用嵌套結構重複一次。

講師常犯三大錯

錯誤後果修正
1. 條件藏在 fallback 位置看不懂觸發條件每個 Sequence 顯式寫 Condition
2. 線性 fallback 硬堆疊成巢狀加一個選項要多一層攤平到同一個 Selector
3. 沒加預設動作所有條件不符時呆站Root Selector 尾端放預設 action

檢查清單：畫完行為樹問自己

  1. 每個 Action 都有自己的 Condition 嗎？（沒靠 fallback 猜）
  2. Root Selector 尾端有預設動作嗎？（不會呆站）
  3. 每一層拿掉會改變語義嗎？（沒改變就扁平化）
  4. 優先順序對嗎？最重要的在最上面嗎？
  5. 加新分支要改到幾個地方？（超過 1 個就是結構有問題）
