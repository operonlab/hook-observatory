PART I

基礎：Tick 怎麼跑、節點怎麼分

CHAPTER 01

Tick — 行為樹的心跳

對應教科書 §2.1 — The Tick-Generation Mechanism

行為樹不是一次執行完的決策，而是一個每週期從根重跑一次的反應式系統。這個週期脈衝叫做 tick。

DEFINITION 1.1

Tick 是由 root 節點向子樹遞迴發出的執行訊號。每個節點收到 tick 後執行一次其邏輯，並回傳 Success / Failure / Running 之一。下個週期 root 會再次發出 tick，樹從根重新評估。

Tick 的傳遞

Tick 從 root 開始，依照 composite 節點的語義決定要把 tick 傳給哪些子節點。例如 Sequence 會從左到右依序 tick 子節點，遇到 Failure 或 Running 就停止並上報。

EXAMPLE 1.1 生活比喻

想像一台心律調節器每秒對身體發送一次「檢查你現在該做什麼」的訊號。訊號從大腦（root）沿著神經（edges）傳到四肢（leaves），每個部位回報「我做好了」「我做不到」「我還在做」— 這就是 tick。

為什麼要每秒重跑？

因為環境會變。上一秒還沒看到敵人，這一秒看到了 — 如果樹只執行一次，小精靈就會繼續吃豆子直到被抓。**重新從 root 評估**讓 BT 能對環境變化做出即時反應（reactivity）。

CHAPTER 03

三種返回值 — S / F / R

對應教科書 §2.1.1 — Return Statuses
DEFINITION 3.1

BT 節點被 tick 後必須回傳以下之一：

  • Success — 任務完成或條件成立
  • Failure — 任務失敗或條件不成立
  • Running — 任務還在進行中，下個 tick 會繼續

為什麼 Running 很重要？

這是 BT 與 Decision Tree 最大的差別。DT 的葉子動作是「瞬間」的，BT 的動作可以跨多個 tick執行。

EXAMPLE 3.1 走路到門口

Action「走到門口」不會在 1 tick 內完成。它會：

  • 第 1 tick：開始走 → 回傳 Running
  • 第 2 tick：還在走 → Running
  • ...
  • 第 N tick：到了 → Success

父 Sequence 收到 Running 就會保持狀態，不會往下 tick 其他子節點，也不會宣告整體成功，等下個週期繼續。

PART II

Composite 節點深入

CHAPTER 04

Sequence -> — AND 邏輯

對應教科書 §1.3.1 — Sequence Node
DEFINITION 4.1

Sequence 按順序 tick 子節點：

  • 子節點回 S → 繼續 tick 下一個
  • 子節點回 F → 立即回傳 F（短路，不 tick 剩下的）
  • 子節點回 R → 立即回傳 R
  • 全部子節點都 S → 回傳 S
EXAMPLE 4.1 開門動作
開門三步驟：前置條件 + 動作
flowchart TD S["-> Open Door"] C1["Door Exists?"] C2["Not Locked?"] A1["Turn Handle"] A2["Push"] S --> C1 S --> C2 S --> A1 S --> A2 classDef cond fill:#fff2cc,stroke:#d6b656,color:#000 classDef act fill:#d3f9d8,stroke:#2f9e44,color:#000 classDef ctrl fill:#f8f9fa,stroke:#868e96,color:#000 class C1,C2 cond class A1,A2 act class S ctrl

任何一步失敗（門被鎖、把手卡住）都會讓整體 Sequence 失敗。這是**前置條件鏈**的標準模式。

CHAPTER 05

Fallback ? — OR 邏輯（也叫 Selector）

對應教科書 §1.3.2 — Fallback Node
DEFINITION 5.1

Fallback 按順序 tick 子節點：

  • 子節點回 S → 立即回傳 S（找到一個能做的就停）
  • 子節點回 F → 繼續試下一個
  • 子節點回 R → 立即回傳 R
  • 全部子節點都 F → 回傳 F
EXAMPLE 5.1 下雨時的應對
優先級救援：最好的方案排最上

Handle Rain"] S1["-> [Has Umbrella?, Open Umbrella]"] S2["-> [Near Shelter?, Run to Shelter]"] S3["-> [Has Hood?, Put Hood On]"] Wet["Get Wet (預設)"] F --> S1 F --> S2 F --> S3 F --> Wet classDef act fill:#d3f9d8,stroke:#2f9e44,color:#000 classDef ctrl fill:#f8f9fa,stroke:#868e96,color:#000 class S1,S2,S3,Wet act class F ctrl
命名注意

Ögren 教科書用 Fallback，Unreal/Unity 業界用 Selector。兩者是同一個東西，本網頁兩種稱呼並用。

CHAPTER 06

Parallel ⇉ — 同時做多件事

對應教科書 §1.3.3 — Parallel Node
DEFINITION 6.1

Parallel (M-of-N) 每個 tick 同時 tick 所有子節點，依照策略判定整體結果：

  • Success-on-all：全部子節點 S 才算 S
  • Success-on-one：任一子節點 S 就算 S
  • M-of-N：至少 M 個 S 就算 S

與 Sequence/Fallback 的「依序 tick」最大差別：Parallel 每個 tick 都 tick 所有子節點。

EXAMPLE 6.1 巡邏機器人
邊走邊監視：Parallel + Success-on-one
flowchart TD P["|| Parallel (success-on-one)"] A1["Patrol Route"] A2["Monitor Enemy"] A3["Check Battery"] P --> A1 P --> A2 P --> A3 classDef act fill:#d3f9d8,stroke:#2f9e44,color:#000 classDef ctrl fill:#fff4e6,stroke:#e67700,color:#000 class A1,A2,A3 act class P ctrl

三個子節點同時運行：巡邏路線、監視敵人、檢查電量。任一發現問題（例如敵人出現 → Monitor 回 S）就中止整個 Parallel，讓上層決定下一步。

⚠ Parallel 的陷阱

Parallel 會同時執行所有子節點的 action — 如果兩個 action 操作同一資源（例如兩個都控制馬達）會衝突。Ögren 教科書強調 Parallel 的子節點最好是互相獨立的感知/動作，而非同時爭搶同一硬體。

CHAPTER 07

Memory 節點的陷阱

對應教科書 §2.2 — Memory vs Memoryless Nodes

很多 BT 函式庫（如 BehaviorTree.CPP、py_trees）提供 Sequence with Memory / Fallback with Memory — 又稱 sticky 版本。Ögren 教科書強烈建議避免使用，本章解釋原因。

DEFINITION 7.1

Memory Sequence：記住上次 tick 到哪個子節點，下個 tick 從那裡繼續，而不是從頭。例如第 1 tick 跑到第 2 個子節點回 R，第 2 tick 會直接從第 2 個子節點繼續。

為什麼誘人？

乍看省事：不用每次從頭評估所有子節點，看似更有效率。

為什麼 Ögren 反對？

🚨 Memory 破壞 Reactivity

假設你有一個 Sequence：-> [Enemy Far?, Walk to Target, Place Bomb]。

  • 第 1 tick：Enemy Far? ✓ → 開始走 → 走到一半（Running）
  • 第 2 tick：如果用 Memory Sequence，直接從 Walk 繼續走，不再檢查 Enemy Far
  • 結果：敵人已經跑到面前，機器人還在傻傻地走去放炸彈

這完全違背了 BT「每 tick 重新評估環境」的 reactivity 原則。

GUIDELINE 7.1

除非你非常確定條件不會在執行中改變（如「檔案讀取」這類與外界無關的內部動作），否則永遠使用 memoryless Sequence/Fallback。需要記住進度的話，把狀態寫到 Blackboard 顯式管理。

PART III

Decorator 與共享資料

CHAPTER 08

Decorator — 修飾單一子節點

對應教科書 §2.2.3 — Decorator Nodes

Decorator 只有一個子節點，任務是轉換或限制子節點的返回值或 tick 頻率。想像它是包在子節點外面的濾鏡。

Decorator行為典型用途
InverterS ↔ F 交換，R 不變把「門開著?」反成「門關著?」
Force Success永遠回 S（不管子節點）讓非關鍵動作失敗也不影響 Sequence
Force Failure永遠回 FDebug 或強制跳過分支
Repeat(N)重複 N 次巡邏 5 圈
Retry(N)失敗時重試最多 N 次網路請求重試
Timeout(T)T 秒後強制回 F防止 action 卡死
Cooldown(T)T 秒內回 F（鎖定）技能冷卻
Limiter(N)最多執行 N 次一次性動作（如開箱子）
EXAMPLE 8.1 Inverter 的用途

BT 標準寫法只有「條件成立 → S」，想表達「條件不成立 → S」就得用 Inverter。

// 原條件節點
Enemy Nearby?           回 S 表示附近有敵人

// 加 Inverter
◇ Inverter
└── Enemy Nearby?       回 S 表示附近「沒有」敵人

CHAPTER 09

Blackboard — 節點之間怎麼傳訊息

對應教科書 §2.4 — Blackboard Architecture
DEFINITION 9.1

Blackboard 是一個 BT 全域可見的 key-value store，節點透過它讀寫共享資料。BT 節點本身無狀態、無返回值資料 — 所有需要跨節點共享的資訊都放 Blackboard。

EXAMPLE 9.1 追擊目標
  • Condition 節點 Find Target：掃描敵人 → 最近者寫到 BB 的 target_id
  • Action 節點 Move to Target：讀 BB 的 target_id → 前往該位置
  • Action 節點 Attack Target：讀 target_id → 攻擊

三個節點完全解耦，但透過 Blackboard 協作完成一個追擊循環。

⚠ Blackboard 的濫用

Blackboard 方便但容易變成「全域變數大垃圾場」。Ögren 建議：

  • Blackboard key 應該有命名前綴（如 perception/target_id）
  • 寫入處應該少，讀取處可以多 — 方便追蹤資料來源
  • 子樹最好有自己的區域 blackboard，避免全域污染
PART IV

設計哲學、比較與模式

CHAPTER 10

BT 的四大設計哲學

對應教科書 §3.1 — Advantages of BTs

1. Reactivity（反應性）

每 tick 從 root 重新評估，對環境變化立即反應。不會被「上次做到哪」綁死。

2. Modularity（模組性）

任何子樹都可以獨立測試、獨立移植。子樹的輸入是 tick、輸出是 S/F/R — 接口極小。

3. Readability（可讀性）

樹狀結構視覺直觀，控制節點少、符號統一。相比 FSM 的箭頭大麻團更易讀。

4. Debuggability（可除錯）

每個節點的返回值可追蹤 — 工具可以即時上色顯示「哪個分支正在 Running」，比 FSM 好除錯太多。

CHAPTER 11

BT vs FSM / HFSM / DT / Subsumption

對應教科書 §4 — Generalization of Earlier Ideas

教科書 Chapter 4 用一整章證明 BT 可以表達其他主流控制架構能表達的所有行為，且更模組化。

與 FSM 的比較

FSM 的痛點

  • 狀態之間的轉換邊會爆炸（O(n²)）
  • 加一個狀態要改多個轉換邊
  • 重用子狀態機困難
  • 視覺上一團亂（「spaghetti transitions」）

BT 的改善

  • 樹狀結構，控制邊線性成長
  • 加一個行為 = 多掛一個子樹
  • 子樹可直接複製到另一棵樹重用
  • 視覺清爽，符號少且統一

五大架構對照

架構核心概念痛點BT 如何改善
FSM狀態 + 轉換邊轉換邊 O(n²) 爆炸樹狀，邊線性成長
HFSMFSM 巢狀父子狀態耦合子樹完全模組化
Decision Tree問題分叉無 Running、無反應性Tick + 三返回值
Subsumption行為層疊優先難擴充、難 debugFallback 取代層疊
Teleo-reactive條件-動作規則只支援單層可合成為 BT
A BT can be designed to act equivalent to any given FSM, HFSM, DT, subsumption architecture, or teleo-reactive program, while providing superior modularity. Colledanchise & Ögren, 2018, §4 summary
CHAPTER 12

PPA Design Pattern — 書中最強設計方法

對應教科書 §3.2 — Postcondition-Precondition-Action Pattern

PPA 是 Ögren 教科書第 3 章提出的目標導向 BT 合成方法。有了 PPA，設計 BT 就不再是憑直覺畫，而是從目標倒推。

DEFINITION 12.1 PPA Template 
目標 G 的 PPA 子樹：

? Goal G
├── Postcondition: G 已達成?   ← 已經達到就不用動
└── -> 達成路徑
    ├── Precondition 1: P₁
    ├── Precondition 2: P₂
    ├── ...
    └── Action A: 執行動作

為什麼 PPA 這麼強？

  1. 目標導向：每個子樹都指向一個明確目標
  2. 自我修正：如果目標已達成（postcondition ✓）就不重複動作
  3. 可組合：要達成 Goal G 需要先達成 P₁，P₁ 本身又可展開成 PPA 子樹，遞迴合成
  4. 可驗證：形式化驗證（書中第 9 章）可檢查 Precondition 是否足以保證 Postcondition

PPA 三層遞迴：拿起物品 = 在附近 + 伸手抓

每一層都是 PPA 結構：先檢查目標是否達成，沒達成就沿著前置條件 → 動作走。第二層目標（Object Reachable）本身又是一個 PPA，遞迴下去。

CHAPTER 13

常見設計陷阱 Top 7

對應教科書 §3.4 — Common Pitfalls
#陷阱症狀修正
1DT-style 巢狀5+ 層 ?/->扁平化到同一 Fallback
2隱含條件Action 裸掛在 Fallback每個 Action 前配明確 Condition
3缺預設動作所有條件不符時樹 return FRoot Fallback 尾端加兜底 action

5Blackboard 污染全域 key 撞名加命名前綴、子樹用局部 BB

7Action 無法回 RunningSequence 跑到一半被中斷長時 action 必須實作 R 狀態
🚨 頭號殺手：DT-style 巢狀

這是新手最常犯的錯 — 把 if-else 決策樹直接套上 BT 符號，產生 5-6 層巢狀結構。KTH 課程第三部分的 "Warrior DT design of BT" 就是作者故意示範的反例，但經常被轉述者誤當成範例教。

判別訣竅：看到超過 3 層的 ? 或 ->，先問自己「這層拿掉語義會變嗎？」— 不變就是冗餘。
